# Project-Specific Agent Instructions

## Code Implementation Style

### Direct Edits (Required)
- **Always edit files directly** - never create `*_example.gd` or `*_template.gd` files
- When providing code, update the actual working files immediately
- Use `replace_string_in_file` or `multi_replace_string_in_file` for code changes
- If a file needs to be created, create it with the final code, not an example

### Documentation Style

#### What to Include
- Step-by-step instructions for user actions
- Code explanations and comments
- "Why this matters" context
- Verification checkpoints

#### What to Avoid
- **No "Troubleshooting" sections** in milestone documentation
- **No "If Something Went Wrong"** sections unless explicitly requested
- **No "Common Problems" subsections**
- Keep documentation focused on the happy path
- Trust the user to ask if they encounter issues
- Don't prioritize updating the IMPLEMENTATION_PLAN or EXTENSIONS_REFERENCE unless explicitly requested. When prompted to fix an issue, the focus should be on thinking properly about the code and providing a solution, not on updating documentation.

### File Organization
- Data-related scripts: `Data/` folder
- Utility classes: `Scripts/Utilities/` folder
- Scene scripts: `Scripts/` folder (direct in root)
- Keep folder structure clean and organized

### Code Quality
- Use data-driven approaches (e.g., IsotopeData singleton)
- Prefer static utility classes for reusable functions
- Add clear comments explaining non-obvious code
- Use `@export` for Inspector-editable properties
- Set sensible defaults

### Godot-Specific
- Always set `sprite.centered = true` for sprites used in positioning systems
- Use `class_name` for utility classes
- Autoload singletons go in `Data/` folder
- Follow Godot naming conventions (snake_case for variables, PascalCase for classes)

## Communication Style
- Be concise
- Focus on what was done, not what could go wrong
- Provide clear next steps
- Don't over-explain edge cases
